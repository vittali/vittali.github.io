Test-driven development (TDD) is a development strategy where the act of testing drives the development process. I always start any kind of development (be it in software or in electronic design) with the development of a test strategy. What can’t be tested doesn’t exist. Unit tests are a specific kind of test that targets the smallest piece of functionality. In my work, this is usually a function. I use unit test frameworks like junit (java) and google-test (cpp) for my desktop development and I want to use a similar workflow for my embedded projects.

This document describes an attempt to set up such a workflow using the Debug Scripting Server DDS that is part of Code Composer Studio CCS from Texas Instruments (TI). I demonstrate how to unit test a function executed on the SimpleLink™ MSP432P4111 high-precision ADC MCU LaunchPad™ Development Kit featuring the MSP432P4111 microcontroller (MCU) from TI.

Related source code can be cloned from the compagnion repo. The code is MCU agnostic, most LaunchPads from TI should work. I use Linux Mint 19.2 Tina, Code Composer Studio 9.2.0, java 12, gradle 5.6.2, Eclipse IDE 2019-09 R (4.13.0).

1. Introduction

When I write java code for desktop applications, I hardly ever use the debugger. And if I feel tempted to launch a debug session, I know that something is wrong with my design. This is, in my view, one of the main benefits of test driven development. If I start a design with testability as an imperative, I usually understand what my code does and I don’t need a debugger. Most importantly, unit testing can be automatized with build tools like gradle or maven. This makes regression testing easy which in turn favors refactoring and modularization efforts. In embedded programming, launching the debugger seems to be the default mode of work and I find this unfortunate. Debugging, in particular stepping through code, is a manual activity. Therefore, refactoring and other code improvements can’t be done with confidence and require additional time and effort.

To bring automatic unit tests to embedded programming, I will explore two different approaches:

  • CCS integrates the Debug Scripting Server (DSS) which allows to load and run programs and to set breakpoints at specific locations. Since DDS has bindings for several desktop programming languages, it can be driven from one of the existing unit test frameworks. I will use java as host language and junit as testing framework.

  • exchange messages between the host and the embedded system under test, for example via a serial link. TI LaunchPads usually offer a serial connection over USB which makes this approach straightforward on any PC or laptop.

Both approaches can be combined. In this document, I focus on DDS, the second approach will be discussed in another memo.

2. Hardware

2.1. Setting the serial number of the debug probe

Since I own several LaunchPads, I allocate unique serial numbers to the debug probe module that is part of most LaunchPads. This allows to load and run programs on several different devices during the same unit test. This scenario will be discussed in a different memo.

I label the debug probe on the LaunchPad with the serial number and verify that it is the only board connected to the host. In this case, I set the serial number to 4

cd $CCS_HOME/ccs/ccs_base/common/uscif/xds110
$ ./xdsdfu -m  (1)
$ ./xdsdfu -s 4 -r (2)
1 set the device into DFU mode (required)
2 set the serial number to 4 and reset the device

3.3. Run the test from the command line with gradle

As mentioned earlier, one of the main benefits of unit tests compared to step-debugging (or sitting in front of an oscilloscope) is the automization of regression testing. Even after seemingly innocuous code modifications, I tend to run one or several tests from the command line.

The code for unit testing with DSS is available for download in the compagnion repo. It is contained in two eclipse projects with gradle nature, which means that all dependencies are handled by gradle and that tests can be run from either the Eclipse IDE or from the command line with gradle:

hotbrix.dss.api.lib

contains implementation independent code, mainly types (interfaces) and constants (enums)

hotbrix.dss.lib

contains implementation dependent code based on the DDS api.

These libaries depend on a various other libraries like softbrix.lib that I will explain in separate memos. The code is based on Functional Programming in Java.

To run the MCU agnostic tests (no need to attach a LaunchPad), I do the following:

$ git clone git@github.com:vittali/vittali.ch-8-mcunit-msp432.git
$ cd vittali.ch-8-mcunit-msp432
$ ./gradlew hotbrix.dss.lib:check  (1)

BUILD SUCCESSFUL in 5s
16 actionable tasks: 16 up-to-date
1 you might need to run gradle wrapper first.

Once you have connected the LaunchPad, I can run the entire test suite:

$ ./gradlew hotbrix.dss.lib:check -PMCU (1)
1 the build.gradle uses the gradle property MCU to control which tests are excluded when no LaunchPad is connected. In particular, this allows to test MCU agnostic functionality on an integration server like Travis.
